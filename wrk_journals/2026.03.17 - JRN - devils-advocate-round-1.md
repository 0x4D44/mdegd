# Devil's Advocate: Round 1

**Date:** 2026-03-17
**Role:** Critical review of project assumptions, design, and risks
**Project state:** Scaffolding only. Tests written for quiz-engine.js (which doesn't exist yet). No src/, no docs/, no questions, no UI. Four days until the birthday.

---

## 0. The Elephant in the Room: Timeline

Before anything else, let's talk about the calendar. Eric's birthday is March 21st. Today is March 17th. That's four days to research ~100 questions, build a quiz engine, create 8+ visual themes, integrate everything, deploy to GitHub Pages, and test it with real humans. This is the single biggest risk to the project.

**Recommendation:** Ruthlessly cut scope now, before you're forced to cut it badly on March 20th at midnight. A polished quiz with 40 good questions and 2 themes will land far better than a rushed quiz with 100 mediocre questions and 8 half-baked themes. See Section 2 for specifics on what to cut.

## 1. User Experience

### 1.1 Is 12 questions the right number?

Twelve is defensible for a "proper" quiz but may be too many for a birthday party context. People at parties have short attention spans, especially after food and drinks. Eight to ten would keep the energy up. Twelve risks the "are we still doing this?" vibe.

However, 12 has the nice property of being divisible by common category counts (2, 3, 4, 6), which makes balanced selection easier. If you stick with 12, consider a natural break point (maybe a birthday-themed interlude at question 6) so it doesn't feel like a slog.

**Recommendation:** Default to 10 questions. This is a birthday gift, not an exam. Faster is funnier. If testing reveals 10 feels too short, bump to 12.

### 1.2 Mobile experience

This will almost certainly be played on phones. Eric will receive a URL via text or WhatsApp. He will tap it on his phone. This is the primary use case, not a secondary one.

Critical mobile concerns:
- **Touch targets:** Quiz answer buttons need to be at least 44x44px (Apple HIG) or 48x48dp (Material). Four-option multiple choice buttons stacked vertically should be fine, but cramming them side-by-side will cause misclicks.
- **Scrolling:** If the question text + 4 options don't fit on one screen without scrolling, you've failed. Eric shouldn't have to scroll to see all the answers. This constrains question length.
- **Text size:** Eric is (presumably) in his 60s-70s. Default body text at 16px is the absolute minimum. Consider 18px for questions and 16px for options.
- **Fat finger problem:** If Eric taps the wrong answer because buttons are too small or too close together, the quiz has betrayed him. Generous padding between options is essential.

**Recommendation:** Design mobile-first. Test on a real phone early. Make answer buttons full-width, stacked vertically, with generous padding. Set a maximum question length (say 120 characters) as a content constraint.

### 1.3 Eric/Ruth mode selection

The name-based difficulty selection is the best joke in the quiz. But it only works if:
1. The player immediately understands that clicking their name starts the quiz.
2. Non-family members who stumble on the URL can still play (they won't know who Eric and Ruth are).
3. The framing makes the joke obvious without explaining it (explaining a joke kills it).

Risks:
- If the landing page just shows two names with no context, a confused user will pick randomly.
- If it over-explains ("Eric mode is really hard! Ruth mode is easier!"), you've ruined the gag.
- If Ruth sees the quiz and doesn't understand why her name means "easy mode," that could go badly. Is Ruth in on the joke? Would she find it funny?

**Recommendation:** Frame it as "Who are you?" with a brief, warm intro: "Welcome to the March 21st Birthday Quiz. Pick your player." Then just the two names as large buttons. The joke reveals itself through play, not through labeling. Maybe add a tiny subtitle under each name if needed (Eric: "The Birthday Boy" / Ruth: "Everyone Else") but keep it minimal.

**Critical question for Arthur:** Will Ruth actually see this? Is she the type to laugh at the difficulty difference, or would she be annoyed? This matters for the framing.

### 1.4 Accidental mode selection

If Eric taps "Ruth" by accident, he's stuck in comedy mode. That's not terrible (the Ruth questions are still fun), but he might want to restart in Eric mode.

Current design has no back button, no restart mechanism, no mode indicator during play.

**Recommendation:** Add a small, unobtrusive "Start over" link in the header or footer. Don't make it prominent (you don't want people quitting mid-quiz) but make it findable. Also show the current player name somewhere ("Playing as: Eric") so they know which mode they're in.

### 1.5 Progress indicator

Yes, absolutely. Without one, at question 7 of 12, the player has no idea if they're nearly done or only halfway. A simple "Question 5 of 12" or a minimal progress bar is essential for pacing.

**Recommendation:** "Question N of M" text, above the question. Simple, clear, zero design overhead.

### 1.6 Accessibility

Let's be honest about the audience: this is a birthday gift for one specific person, probably shared with a handful of family members. Full WCAG 2.1 AA compliance would be gold-plating.

That said, some basics cost nothing:
- **Color contrast:** Ensure text is readable against all 8+ random themes. This is where random themes create real risk — one bad theme combination and the quiz is unreadable. Every theme needs contrast-checking.
- **Keyboard navigation:** Works for free if you use semantic HTML (`<button>` elements, not styled `<div>`s). Costs nothing. Do it.
- **Screen readers:** Use proper heading hierarchy and ARIA labels on interactive elements. Again, near-zero cost with semantic HTML.
- **Motion/animation:** If themes include animations, respect `prefers-reduced-motion`.

**Recommendation:** Use semantic HTML elements. Test each theme for contrast. Don't over-invest in accessibility for a one-time birthday gift, but don't actively break it either.

## 2. Quiz Design

### 2.1 Replay value and question pool size

With 100 questions and 12 per play, a player would see 12% of the pool per play and need ~8-9 plays to see most questions (with increasing overlap). Is that enough replay value?

For the actual use case — a birthday party and maybe a few replays in the following days — this is massive overkill. Nobody is going to play this quiz 8 times. The realistic scenario:
- Eric plays once in Eric mode.
- Ruth (or someone else) plays once in Ruth mode.
- Maybe 2-3 other family members have a go.
- Eric might replay once more the next day.

That's 3-5 plays, max. You could get away with 40-50 questions and nobody would ever notice repetition.

**This matters because of the timeline.** Writing 100 high-quality questions with well-crafted Eric (hard) and Ruth (easy/funny) answer sets is a LOT of content work. Each question needs:
- A factually correct claim about March 21st
- 4 plausible but wrong answers for Eric mode (hard)
- 4 comedically wrong answers for Ruth mode (easy)
- An explanation
- Category tagging

That's roughly 100 x 10 individual pieces of content = ~1000 items to write and fact-check. In four days. While also building the app.

**Recommendation:** Cut to 50 questions. This gives 4+ unique plays with 12 questions each, which is more than enough. Invest the saved time in question quality and fact-checking. A smaller pool of excellent questions beats a larger pool of mediocre ones.

### 2.2 Category balance

The current `selectQuestions` function (per the tests) does pure random selection. This means you could theoretically get 12 US state history questions and zero bridge questions. That's a poor experience.

**Recommendation:** Implement category-aware selection. The `getCategoryDistribution` function already exists in the test file, suggesting this was considered. Ensure each play includes questions from at least 3-4 different categories. A simple approach: select 2-3 questions from each category, then fill remaining slots randomly.

### 2.3 The "all answers are the same" gag

This is the signature Ruth-mode joke. If every Ruth question has "all the same" answers, it becomes predictable by question 3 and stops being funny. If it only appears once, it might be missed.

**Recommendation:** Use it for exactly 2-3 out of 12 questions. Enough to be a running gag, not so much that it's the entire bit. Space them out — don't put them back-to-back. The first one should land around question 3-4 (by then Ruth is paying attention but hasn't settled into a pattern yet). The rest of the Ruth questions should have one obviously correct answer among silly wrong answers.

### 2.4 Timer

No. Absolutely not. A timer would make Eric anxious and Ruth confused. This is a birthday gift, not a game show. The fun is in reading the questions and answers, not in speed.

### 2.5 Difficulty progression

For Eric mode: slight difficulty increase could work, building to a satisfying challenge. But don't over-engineer this — random ordering is fine for a party quiz.

For Ruth mode: definitely not. The comedy works better when the "difficulty" is consistently absurd.

**Recommendation:** Random ordering. Don't spend development time on difficulty curves.

## 3. Content Risks

### 3.1 Fact-checking

This is the highest-stakes content risk. The kickoff journal lists events like the Nevada Enabling Act (1864) and Alcatraz closing (1963). These need to be bulletproof.

Specific concerns:
- **Date ambiguity:** Some events span multiple days. "Selma to Montgomery marches" — the marches happened on multiple dates. Which specific event occurred on March 21st? If Eric knows it was actually March 25th (arrival in Montgomery) and the quiz says March 21st, you've lost credibility.
- **Enabling Act vs. Admission:** Nevada's Enabling Act was March 21, 1864, but statehood was October 31, 1864. If a question conflates these, Eric will catch it.
- **First tweet:** Jack Dorsey's first tweet was March 21, 2006. This one is well-documented and safe.
- **Napoleonic Code:** Enacted March 21, 1804. Solid.

**Recommendation:** Every factual claim must be verified against at least two independent sources. The questions themselves should be precisely worded — "On March 21, 1864, what act did Congress pass regarding Nevada?" not "When did Nevada become a state?" Create a fact-check column in whatever spreadsheet or data structure holds the questions, and mark each one verified.

### 3.2 Sharpeville Massacre

The Sharpeville Massacre (March 21, 1960) is one of the most significant events in the anti-apartheid struggle. 69 people were killed by police. It's now the International Day for the Elimination of Racial Discrimination.

Including it in a birthday quiz is... complicated. On one hand, it's genuinely important history and Eric (if he's a history buff) would expect it. On the other hand, a multiple-choice birthday quiz trivializes serious historical suffering in a way that might feel uncomfortable.

Consider the context: Eric opens his birthday quiz, answers a fun question about Nevada, then gets "On March 21, 1960, how many people were killed by South African police at Sharpeville?" That tonal whiplash could kill the mood.

**Recommendation:** Include it, but carefully. Frame the question around the legacy (e.g., "March 21st is the International Day for the Elimination of Racial Discrimination. What 1960 event prompted the UN to establish this day?"). This focuses on the positive legacy rather than the death toll. Avoid questions that reduce it to a trivia fact about body counts.

Alternatively: skip it entirely and use the International Day framing as the question itself, without naming the massacre directly in the question text. Let the explanation provide the full context.

### 3.3 Bridge and card cheating connections

The kickoff journal explicitly acknowledges there are "no direct March 21st events for bridge, card cheating, NSHEB." The plan is "creative questions, family jokes, stretch connections."

This is a risk. If the "stretch" is too obvious, the questions will feel forced and Eric will notice. "What card game was being played somewhere on March 21st?" is not a real question.

Better approaches:
- **Meta-questions:** "Eric's favourite card game, bridge, requires exactly how many players?" This isn't about March 21st at all, but it's about Eric and it's his birthday quiz. That's allowed.
- **Family in-jokes:** These don't need a March 21st connection. They need an Eric connection.
- **Honest framing:** A question could openly acknowledge the stretch: "We couldn't find any March 21st bridge events, so here's one anyway..."

**Recommendation:** Don't force March 21st connections where they don't exist. The quiz is "for Eric's birthday on March 21st," not "every question must be about March 21st." A mix of March 21st history + Eric-specific personal questions would be more fun than 100% March 21st with some tortured bridge connections.

### 3.4 NSHEB questions

The North of Scotland Hydro-Electric Board was renamed Scottish Hydro-Electric in 1991 and has gone through several further corporate iterations. These questions could be brilliant (if Eric genuinely has deep knowledge here) or completely baffling (if his interest is casual).

**Question for Arthur:** What does Eric actually know about NSHEB? Did he work there? Does he live in the area? Is this a deep expertise or a passing interest? The answer determines whether NSHEB questions should be hard-but-fair or playful-and-light.

**Recommendation:** Start with 2-3 NSHEB questions and test them on Arthur first. If Arthur (who knows Eric) can't gauge the difficulty, the questions are probably too obscure.

## 4. Technical

### 4.1 GitHub Pages deployment

GitHub Pages can serve from the root of the `main` branch, the `/docs` folder on `main`, or a `gh-pages` branch. The project currently has a `/docs` folder (empty) alongside `/src`.

**Recommendation:** Use `/docs` on `main`. This keeps the deployable site separate from the source/test files, and it's the simplest GitHub Pages configuration. The `/docs` folder would contain `index.html`, CSS, and JS. No build step needed — just copy the final files there.

However, this creates a workflow question: does `/src` contain the "real" source that gets bundled into `/docs`? Or is `/docs` the only source? If there's no build step, why have `/src` at all? The test file imports from `../src/quiz-engine.js`, suggesting `/src` is the source of truth and something needs to copy/transform it into `/docs`.

**Recommendation:** Clarify the build/deploy story now. Options:
1. **No build step, source IS the deployment:** Put everything in `/docs`, test against `/docs` files. Simple but means tests live alongside production code.
2. **Minimal copy step:** Develop in `/src`, have a script that copies to `/docs`. Adds a step but keeps things clean.
3. **Just use root:** Configure GitHub Pages to serve from root. Put `index.html` at root. Tests can import from `./src/`. Simplest possible approach.

Given the four-day timeline, option 3 is the pragmatic choice.

### 4.2 Source code visibility

The question pool will be in a JavaScript file, visible to anyone who views source or opens DevTools. Eric could cheat.

Does this matter? Realistically, no. Eric is not going to view-source on his birthday quiz. And if he does, that's its own kind of entertainment.

But consider: if the URL gets shared more widely (family group chat, social media), someone might spoil answers. This is unlikely to matter for a one-time birthday event.

**Recommendation:** Don't worry about it. If you wanted to mitigate, you could base64-encode the answers (not encryption, just obfuscation), but this adds complexity for zero real benefit. Ship it in plaintext.

### 4.3 Offline support

The kickoff journal says "No dependencies, works offline." But it doesn't mention a service worker.

Without a service worker, "works offline" means "works offline if you've already loaded it and haven't closed the tab." The moment you close the tab or clear cache, you need the network again.

**Recommendation:** Skip the service worker. This is a birthday quiz that will be used for one day. If there's no internet at the party location, download the page beforehand. A service worker adds complexity that isn't justified for this use case.

### 4.4 Share results

A "share your score" feature (copy to clipboard or screenshot-friendly result card) would be nice for family group chats. Imagine: Eric screenshots his 4/12 score and Ruth screenshots her 12/12 score. That's the joke paying off.

**Recommendation:** Add a simple, clean results screen that looks good as a screenshot. Consider a "Copy result" button that generates text like "I scored 8/12 on Eric's March 21st Birthday Quiz (Eric mode)! Try it: [URL]". Low effort, high social value.

### 4.5 Testing infrastructure

The test file imports from `../src/quiz-engine.js`, which doesn't exist yet. The vitest config uses jsdom environment, suggesting DOM testing is planned. Good foundation.

One concern: the tests for `getResultMessage` only check that a string is returned and that Eric/Ruth messages differ. They don't test the actual content. This means the result messages could say anything and the tests would pass. That's fine for now (the messages are creative content, not logic), but consider adding at least one test that checks for a specific keyword or tone marker.

## 5. What Are We Missing?

### 5.1 Birthday message

**Yes.** The results screen should include a birthday message. This is a birthday gift. If the quiz ends with just "You scored 7/12" and nothing else, it's a quiz, not a gift. A warm, personal message at the end (even just "Happy Birthday, Dad!") transforms this from "a thing Arthur made" to "a thing Arthur made *for Eric*."

**Recommendation:** End screen includes a birthday message. Different for Eric mode ("Happy Birthday! Now you know what it feels like to be tested on obscure trivia") and Ruth mode ("Happy Birthday, Eric! Ruth says hi"). Make it personal.

### 5.2 Sound effects

No. Sound auto-playing on a webpage is hostile UX. Someone will be playing this in a quiet room or at a party where music is already playing. Sound adds nothing here.

### 5.3 Group/projected play

This is actually a great idea and should influence the design. If the quiz could be projected on a TV or screen at the party, with people shouting out answers, that's a much more fun party activity than individual phone play.

Implications for design:
- **Font sizes need to be large enough to read from across a room.** This aligns with the mobile-first recommendation anyway.
- **No scrolling per question.** Everything visible at once.
- **Maybe a "reveal answer" button** rather than auto-advancing after selection. This lets the group discuss before revealing.

**Recommendation:** Design as if it will be projected. Large text, high contrast, no scrolling, clear "next" actions. This also makes the phone experience better, so there's no downside.

### 5.4 Score tracking (localStorage)

Overkill for this use case. localStorage persistence adds code for a feature that matters only if people replay multiple times over multiple days. They won't.

**Recommendation:** Skip it. Show the score at the end, done.

### 5.5 Ruth mode question count

Interesting idea. If Ruth mode is comedy-easy, 12 questions of "obviously correct" might get repetitive. But fewer questions means less comedy content.

**Recommendation:** Same count for both modes. The Ruth questions need to be entertaining enough to sustain the count. If they're not, that's a content quality problem, not a quantity problem.

### 5.6 Easter eggs

The "card up the sleeve" idea has potential. Maybe: if someone gets 12/12 in Eric mode (which should be extremely difficult), reveal a hidden animation or message. This rewards genuine expertise and gives Eric a goal if he replays.

Other ideas:
- A Konami code or hidden tap sequence that reveals something.
- If the date is actually March 21st when played, an extra-special birthday banner.
- If played in Ruth mode, the final score is always reported as "12/12 - Perfect!" regardless of actual score.

**Recommendation:** One easter egg, maximum. The date-detection one (special message if played on March 21st) is the most elegant because it requires zero user action and rewards playing on the actual birthday.

## 6. Assumptions We're Making

### 6.1 That Eric will find the difficulty difference funny

This is the core assumption of the entire project. If Eric interprets "Ruth mode is easy" as "Arthur thinks Mum is stupid," the quiz is a disaster.

**Mitigation:** The framing must be affectionate, not condescending. Ruth mode isn't "easy" — it's "fun and ridiculous." The comedy comes from the absurd wrong answers, not from the implication that Ruth needs easier questions. If anything, Ruth mode should feel like Ruth is in on the joke.

**Question for Arthur:** Has Eric/Ruth seen this kind of joke before in the family? Is "Eric knows obscure facts and Ruth thinks that's ridiculous" an established family dynamic? If yes, this works. If not, tread carefully.

### 6.2 That 100 questions is enough variety

See Section 2.1. 100 is more than enough — it's probably too many given the timeline. Cut to 50.

### 6.3 That March 21st has enough interesting events

The initial research found strong events for general history and US history. But the gaps in bridge, card cheating, and NSHEB are a real problem (see 3.3). Stretching March 21st to cover all topic areas will produce weak questions.

**Recommendation:** Reframe the quiz. It's "Eric's Birthday Quiz" that happens to be on March 21st. Some questions are about March 21st events. Some are about Eric personally. Some are about his interests. The March 21st connection is a theme, not a rule.

### 6.4 That a web page is the right medium

Alternatives considered:
- **Printed cards:** More tactile, no tech failures, works without internet. But can't randomize, can't do the Eric/Ruth mode switch elegantly, can't reshare.
- **Slideshow (PowerPoint/Google Slides):** Easy to project, familiar format. But static, no interactivity, no scoring.
- **Kahoot or similar:** Already built, handles projection and multiplayer. But branded, requires accounts, less personal.

A custom web page is the right choice because it supports all the features (randomization, modes, themes, scoring, shareability) and is a more personal gift. The risk is that it's more work than alternatives, which brings us back to the timeline.

### 6.5 That the URL will be easy to share

A GitHub Pages URL will be something like `https://0x4d44.github.io/mdegd/` (or whatever the repo name is). That's not the friendliest URL.

**Recommendation:** Either:
1. Accept the GitHub URL and share it as a link (nobody types URLs anymore).
2. If a custom domain is available, configure it. But that's extra work for a birthday gift.
3. Use a URL shortener to create something memorable.

The GitHub URL is fine. Share it as a clickable link in a message. Don't overthink this.

## Summary

The concept is strong. The Eric/Ruth mode split is genuinely funny. The March 21st theme gives it structure. The main risks are:

1. **Timeline.** Four days is tight. Cut scope aggressively (50 questions, 2-3 themes).
2. **Content quality over quantity.** One wrong fact will be remembered longer than 100 right ones.
3. **Sensitive content.** Sharpeville needs careful framing.
4. **Mobile-first.** This will be played on phones. Design accordingly.
5. **The Ruth question.** Make sure the joke lands with warmth, not condescension.

The quiz engine tests are a good foundation. The TDD approach is right. Now it's about executing quickly on the content and keeping the scope honest.
